# Glossary — Implementing Domain-Driven Design

**A+ES** — Event Sourcing used as the persistence mechanism for Aggregates (App A)
**Adapter** — Translates a client protocol or foreign representation into the application's own API (Ch 4, Ch 13)
**Aggregate** — A transactional consistency boundary: a Root Entity plus the parts that must stay immediately consistent (Ch 10)
**Almost-infinite scalability** — Scale via continuous repartitioning of Aggregate storage, enabled by identity-only references (Ch 10)
**Anemic Domain Model** — Objects of getters and setters with no behavior; logic pushed into Services (Ch 1, Ch 7)
**Anticorruption Layer (ACL)** — Downstream defensive translation: Separated Interface → Adapter → Translator → local type (Ch 3, Ch 13)
**Application Service** — Thin, transaction- and security-owning coordinator of one use case flow; holds no business logic (Ch 4, Ch 7, Ch 14)
**Archived log** — An immutable, permanently cacheable NotificationLog page (Ch 8)
**Assessment View** — Subdomains overlaid on a messy existing system to reason about it (Ch 2)
**Big Ball of Mud** — Mixed models with inconsistent, unenforced boundaries (Ch 2, Ch 3)
**BOTE calculation** — Back-of-the-envelope count of objects an Aggregate really holds at production scale (Ch 10)
**Bounded Context** — A conceptual boundary within which a Ubiquitous Language has specific meaning (Ch 2)
**Collection-Oriented Repository** — Set-mimicking interface (`add`/`remove`, no `save`) for change-tracking persistence (Ch 12)
**Command Handler** — An Application Service method invoked by a serialized, temporally decoupled Command (Ch 14, App A)
**Command model** — CQRS write side: command-only Aggregates, no getters (Ch 4)
**Conformist** — Downstream adopts the upstream model wholesale, with no translation (Ch 3)
**Consistency boundary** — The line inside which invariants always hold; consistency outside it is irrelevant (Ch 10)
**Context Map** — A diagram plus the actual source integrations between Bounded Contexts (Ch 3)
**Core Domain** — The Subdomain where the business must excel; deserves the best people and tactical patterns (Ch 1, Ch 2)
**CQRS** — Command-Query Responsibility Segregation: separate write and read models (Ch 4)
**Customer-Supplier** — Upstream plans around a genuinely negotiating downstream (Ch 3)
**Data Transformer** — Client-supplied double-dispatch object producing an output format (Ch 14)
**DDD Scorecard** — Weighted project scoring; total ≥ 7 means seriously consider DDD (Ch 1)
**DDD-Lite** — Tactical patterns applied without strategic design (Ch 1)
**Deferred Validation** — Complex validation postponed to the last responsible moment, collecting all failures (Ch 5)
**Dependency Inversion Principle (DIP)** — Both layers depend on abstractions; Repository interfaces live in the model (Ch 4)
**Disconnected Domain Model** — Using a Repository from inside an Aggregate; a less favorable lazy-load form (Ch 10)
**Domain Event** — An immutable, past-tense record of an occurrence the domain experts care about (Ch 8)
**Domain Expert** — Someone who knows the business; not a job title (Ch 1)
**Domain Payload Object (DPO)** — Carries whole Aggregate instances to the UI within a single VM (Ch 14)
**Domain Service** — A stateless domain operation, named in the Ubiquitous Language, that fits no single Entity or Value (Ch 7)
**Double-dispatch** — Passing a Domain Service into an Aggregate command method so the Aggregate calls back (Ch 10)
**Early identity generation** — Identity assigned before persistence; required when Events publish at construction (Ch 5)
**Event Store** — Append-only, strongly consistent persistence of serialized Events per Stream (Ch 8, App A)
**Event Stream** — The append-only ordered Event list for one Aggregate identity (App A)
**Events enrichment** — Adding non-essential members to an Event to simplify consumers (App A)
**Eventual Consistency** — A modeled, expert-negotiated delay between Aggregates; not a bug (Ch 3, Ch 10)
**Factory Method on Aggregate Root** — Language-named creation behavior on an existing Root (Ch 11)
**False invariant** — A developer-imposed constraint mistaken for a business rule; drives large clusters (Ch 10)
**Focused Aggregate** — A small Aggregate built around a single behavioral aspect (App A)
**Generic Subdomain** — Required by the business but capturing nothing special; buy or outsource (Ch 2)
**Given-When-Expect** — Event-sourced specification: past Events, invocation, expected Events (App A)
**Guard** — A precondition assertion that throws rather than admit invalid state (Ch 5, Ch 6)
**HATEOAS** — Server-embedded links drive client state transitions (Ch 4)
**Hexagonal Architecture** — Ports and Adapters; the model sits inside, all protocols outside (Ch 4)
**Idempotent operation** — Repeated execution produces no additional effect (Ch 8, Ch 13)
**Intention-Revealing Interface** — Operations named from the domain experts' verbs (Ch 5)
**Invariant** — A business rule that must always be transactionally consistent (Ch 10)
**Large-cluster Aggregate** — Designed for compositional convenience; collides under optimistic concurrency (Ch 10)
**Layer Supertype** — An abstract base carrying cross-cutting model concerns, e.g. hiding a surrogate key (Ch 5, Ch 6)
**Long-Running Process (Saga)** — A distributed process with its own identity and a completion state tracker (Ch 4, Ch 13)
**Module** — A named container of highly cohesive domain objects; a Java package or C# namespace (Ch 9)
**Notification** — An envelope carrying an Event with typeName, version, and occurredOn (Ch 8, Ch 13)
**NotificationLog** — A fixed-size current or archived page of Notifications published as a RESTful resource (Ch 8)
**Object schizophrenia** — Delegates ignorant of the originating object's identity (Ch 5)
**Open Host Service (OHS)** — A published protocol expanded to serve new integrators (Ch 3, Ch 13)
**Optimistic concurrency** — A version checked before save; a stale version is rejected (Ch 10, App A)
**Output Port** — A named write target with a per-client adapter, used by void Application Services (Ch 14)
**Partnership** — Two teams succeed or fail together, with coordinated planning (Ch 3)
**Persistence-Oriented Repository** — Explicit `save()` on both create and modify (Ch 12)
**Presentation Model** — A bidirectional Adapter deriving view properties and tracking user edits (Ch 14)
**Problem Space** — The Core Domain plus the Subdomains it uses (Ch 2)
**Process state tracker** — An Aggregate holding a Process identity and completion flags (Ch 4, Ch 13)
**Published Language (PL)** — A documented shared exchange language, often a custom media type (Ch 3, Ch 13)
**Query model** — CQRS read side: a denormalized store shaped per UI view (Ch 4)
**Read Model Projection** — A disposable, Event-driven, rebuildable persistent view (App A)
**Repository** — Collection-like access to whole Aggregates; one per Aggregate type (Ch 12)
**Segregated Core** — Refactoring a supporting capability out of the Core Domain (Ch 2)
**Separated Interface** — Interface declared in the model, implementation supplied elsewhere (Ch 7)
**Shared Kernel** — A shared model subset with joint ownership; changes require agreement (Ch 3)
**Side-Effect-Free Function** — Produces output without modifying its own state (Ch 6)
**Snapshot** — Serialized Aggregate state at a known version, bounding Event replay (Ch 4, App A)
**Solution Space** — The one or more Bounded Contexts that implement the Problem Space (Ch 2)
**Standard Type** — A descriptive Value distinguishing types of things (Ch 6)
**Strategic Design** — Deciding what to invest in and who is involved: Contexts, Maps, Language (Ch 1, Ch 2)
**Subdomain** — One logical business function; a problem-space tool (Ch 2)
**Supporting Subdomain** — Essential and specialized, but not the Core (Ch 2)
**Surrogate identity** — An ORM-only identity hidden from the domain (Ch 5)
**Tactical Design** — Aggregates, Entities, Value Objects, Services, Events (Ch 1)
**Tell, Don't Ask** — Permits navigation for queries but keeps mutation inside the Aggregate (Ch 10)
**Translation Map** — A representation-to-Value-Object mapping diagram (Ch 3)
**Ubiquitous Language** — The team's rigorous, shared language, applicable within one Bounded Context (Ch 1)
**Use case optimal query** — A complex Repository query projected directly into a Value Object (Ch 12, Ch 14)
**User-aggregate affinity** — Only one user works a given set of Aggregate instances at a time (Ch 10)
**Value Object** — An immutable concept that measures, quantifies, or describes; compared by value (Ch 6)
**Whole Value** — Related attributes forming one integral measure (Ch 6)
