# Glossary — Domain-Driven Design

**ABSTRACT CORE** — the most fundamental concepts factored into their own MODULE of interfaces/abstract classes expressing most cross-subdomain interaction (Ch 15)
**Account / Entry** — value held, plus the immutable record of each change to it; entries are never removed (Ch 11)
**Accrual** — accounting for income or expense at the time it is incurred, regardless of when money changes hands (Ch 9, Ch 11)
**ADAPTER** — wrapper letting a client use a different protocol than the implementer understands; a component of an ANTICORRUPTION LAYER (Ch 14)
**AGGREGATE** — a cluster of associated objects treated as a unit for data changes, with a root and a boundary (Ch 6, Ch 7)
**Analysis model** — a model built to understand a business with no consideration of its role in software; Evans' chief target of criticism (Ch 3)
**Analysis pattern** — a group of concepts representing a common construction in business modeling (Fowler 1997) (Ch 11)
**ANTICORRUPTION LAYER** — an isolating layer providing clients functionality in terms of their own model, translating both directions to a foreign system (Ch 7, Ch 14)
**Architectural framework** — intrusive infrastructure dictating how other layers are implemented (Ch 4)
**ASSERTION** — a stated post-condition, precondition, or class/AGGREGATE invariant describing state rather than procedure (Ch 10)
**BOUNDED CONTEXT** — the explicitly bounded part of a system within which a single model applies and is kept unified (Ch 14)
**Breakthrough** — an abrupt, high-value model shift where versatility rises as complexity evaporates (Ch 8)
**Capability / Potential layer** — *what could be done?* — the resources enabling the business (Ch 16)
**CLOSURE OF OPERATIONS** — an operation whose return type matches the type of its argument(s) and implementer (Ch 10)
**COHESIVE MECHANISM** — a conceptually coherent computational framework extracted behind an INTENTION-REVEALING INTERFACE (Ch 15)
**Commitment layer** — *what have we promised?* — goals directing future operations that emerge from ongoing business (Ch 16)
**COMPOSITE** — uniform whole-part hierarchy where parts and wholes share a conceptual type (Ch 10, Ch 12)
**CONCEPTUAL CONTOUR** — the underlying domain division that explains where a design shears under change (Ch 10)
**CONFORMIST** — slavishly adhering to an upstream model to eliminate translation, when upstream won't collaborate (Ch 14)
**Conservation / double-entry bookkeeping** — every credit has a matching debit, within a closed system (Ch 11)
**CONTEXT MAP** — a global view naming every BOUNDED CONTEXT and describing the points of contact between them (Ch 14)
**CONTINUOUS INTEGRATION** — frequent merging of code *and* relentless exercise of the UBIQUITOUS LANGUAGE, within one context (Ch 14)
**Continuous learning** — deliberately growing domain and modeling knowledge over a project's life (Ch 1)
**CORE DOMAIN** — the parts of the model distinctive and central to the application's purpose; where most value should be added (Ch 15)
**CUSTOMER/SUPPLIER DEVELOPMENT TEAMS** — a formalized upstream/downstream relationship where downstream plays the customer in planning (Ch 14)
**Declarative design / declarative style** — writing part of a program as an executable specification, or achieving most of that benefit through combinable, meaning-carrying elements (Ch 10)
**Deep model** — a lucid expression of the experts' primary concerns that sloughs off superficial aspects of the domain (Ch 1, Ch 8)
**Decision Support layer** — *what action should be taken?* — analysis and decision making atop Operations (Ch 16)
**Denormalization** — storing multiple copies of the same data so it clusters with each owning ENTITY (Ch 5)
**Distillation document** — 3–7 sparse pages describing the CORE and its primary interactions; also a change-governance trigger (Ch 15)
**DOMAIN VISION STATEMENT** — a ~1 page value proposition for the CORE DOMAIN, ignoring what doesn't distinguish it (Ch 15)
**Duplicate concepts** — two model elements representing the same concept, requiring dual maintenance (Ch 14)
**ENTITY (Reference Object)** — an object defined primarily by identity and continuity through a life cycle (Ch 5)
**ENTERPRISE SEGMENT** — a set of dimensions defining a way of breaking down a business (Fowler 1996) (Ch 7)
**EVOLVING ORDER** — letting the large-scale structure evolve with the application, possibly changing type entirely (Ch 16)
**Explanatory model** — a model used only to teach the domain, deliberately not in UML, never driving design (Ch 2)
**FACADE** — a simplified alternative interface to a subsystem, written in *that* subsystem's model (Ch 14)
**FACTORY** — a program element whose responsibility is atomic, invariant-enforcing creation of complex objects and AGGREGATES (Ch 6)
**False cognate** — two people using the same term believing they mean the same thing when they don't; the most insidious splinter (Ch 14)
**Firing modes** — eager / Account-based / Posting-Rule-based initiation of Posting Rules (Ch 11)
**FLYWEIGHT** — sharing one immutable instance for many identical VALUE OBJECTS; a purely implementational pattern (Ch 5, Ch 12)
**GENERIC SUBDOMAIN** — a cohesive subdomain that isn't your project's motivation, modeled with no trace of your specialties (Ch 15)
**HANDS-ON MODELERS** — the rule that anyone contributing to the model must touch code, and anyone changing code must express the model in it (Ch 3)
**HIGHLIGHTED CORE** — making the CORE effortlessly visible via a distillation document and/or flagged elements (Ch 15)
**INTENTION-REVEALING INTERFACE** — names describing effect and purpose, never the means (Ch 10)
**Invariant** — a consistency rule that must hold whenever data changes (Ch 6)
**KNOWLEDGE LEVEL** — a group of objects describing how another group should behave; REFLECTION applied to the domain layer (Ch 16)
**Knowledge crunching** — collaborative distillation of domain information into abstractions carrying the business's principles (Ch 1)
**LAYERED ARCHITECTURE** — UI / Application / Domain / Infrastructure, each depending only on layers beneath (Ch 4)
**Local identity** — identity distinguishable only within an AGGREGATE (Ch 6)
**MODEL-DRIVEN DESIGN** — a single model literally driving both analysis and implementation (Ch 3)
**Modeling out loud** — refining a model by speaking scenarios in it; rough edges are easy to hear (Ch 2)
**MODULE (Package)** — a named grouping of model concepts telling part of the domain's story (Ch 5)
**Operations layer** — *what is being done?* — activities as they actually are, not as we want them (Ch 16)
**OPEN HOST SERVICE** — an open protocol exposing your subsystem as SERVICES to many consumers (Ch 14)
**PLUGGABLE COMPONENT FRAMEWORK** — an ABSTRACT CORE of interfaces plus a hub allowing diverse implementations to substitute freely (Ch 16)
**Policy** — a domain-meaningful name for STRATEGY when meaning, not substitutability, motivates it (Ch 1, Ch 12)
**Policy layer** — *what are the rules and goals?* — mostly passive constraints on other layers (Ch 16)
**Posting Rule** — an explicit object deriving an Entry in an output Account from an Entry in an input Account (Ch 11)
**PUBLISHED LANGUAGE** — a well-documented shared interchange language, never equated with the host's model (Ch 14)
**Reconstitution** — creating an instance from stored data; conceptually mid-life-cycle, not creation (Ch 6)
**REPOSITORY** — an object providing the illusion of an in-memory collection of all objects of a type (Ch 6)
**RESPONSIBILITY LAYERS** — layers defined around broad domain responsibilities that tell the system's story (Ch 16)
**SEGREGATED CORE** — the CORE physically repackaged into its own MODULES, strengthening its cohesion (Ch 15)
**SEPARATE WAYS** — declaring a BOUNDED CONTEXT to have no connection to others at all (Ch 14)
**SERVICE** — a stateless standalone operation in the model, named for an activity, encapsulating no state (Ch 5)
**SHARED KERNEL** — a subset of the model both teams agree to share, changed only by consultation (Ch 14)
**SIDE-EFFECT-FREE FUNCTION** — an operation returning a result with no observable side effects (Ch 10)
**SMART UI** — business logic in the UI, integration through a shared database; a legitimate pattern, and DDD's opposite fork (Ch 4)
**SPECIFICATION** — a predicate-like VALUE OBJECT determining whether an object satisfies some criteria (Ch 9, Ch 10)
**STANDALONE CLASS** — a class understandable entirely alone apart from primitives; the extreme of low coupling (Ch 10)
**STRATEGY** — a family of encapsulated, interchangeable algorithms; as a domain pattern, an expressed policy or process (Ch 12)
**Subsumption** — one SPECIFICATION being strictly more stringent than another; logical implication (Ch 10)
**SYSTEM METAPHOR** — a concrete analogy adopted as a large-scale structure and absorbed into the LANGUAGE (Ch 16)
**UBIQUITOUS LANGUAGE** — the shared model-based language used in all team communication and in the code (Ch 2)
**Unification** — internal consistency of a model: unambiguous terms, no contradictory rules (Ch 14)
**VALUE OBJECT** — an object representing a descriptive aspect of the domain with no conceptual identity; immutable by default (Ch 5)
**WHOLE VALUE** — Ward Cunningham's pattern: a VALUE's attributes must form a conceptual whole (Ch 5)
