# Glossary — Design Patterns (GoF)

Terms from Appendix A plus the catalog's recurring role names. Chapter refs in parens.

**Abstract class** — a class whose primary purpose is to define an interface; defers some or all implementation to subclasses and cannot be instantiated. (Ch 1)
**Abstract coupling** — class A holds a reference to *abstract* class B, so A depends on a type, not a concrete class. The mechanism behind most of the catalog's flexibility. (Ch 1)
**Abstract operation** — declares a signature but no implementation; a C++ pure virtual member function. (Ch 1)
**Acquaintance relationship** — a class merely *refers to* another ("uses-a"); weaker and looser than aggregation. (Ch 1)
**Aggregate object** — an object composed of subobjects (its *parts*) for which it is responsible. (Ch 1, Iterator)
**Aggregation relationship** — the "owns-a / is-responsible-for" relationship between an aggregate and its parts. (Ch 1)
**Black-box reuse** — reuse via object composition; composed objects reveal no internal details. (Ch 1, Ch 6)
**Class** — defines an object's interface *and* implementation: internal representation plus operations. (Ch 1)
**Class diagram** — depicts classes, their internal structure and operations, and static relationships. (Ch 1, App B)
**Class operation** — targeted to a class, not an instance; a C++ static member function. (Ch 1, Singleton)
**Concrete class** — has no abstract operations; can be instantiated. (Ch 1)
**Constructor / Destructor** — C++ operations invoked automatically to initialize a new instance / finalize one about to be deleted. (Ch 1)
**Coupling** — the degree to which software components depend on each other. (Ch 1)
**Delegation** — an object forwards a request to a delegate that carries it out on its behalf; composition's answer to inheritance. (Ch 1, State, Strategy, Bridge)
**Design pattern** — systematically names, motivates, and explains a general design for a recurring problem: problem, solution, when to apply, consequences, plus implementation hints and examples. (Ch 1)
**Double dispatch** — the operation executed depends on the run-time types of *two* objects (element and visitor). (Visitor)
**Dynamic binding** — run-time association of a request to an object and one of its operations; in C++, only virtual functions. (Ch 1)
**Encapsulation** — hiding representation and implementation inside an object; operations are the only access path. (Ch 1)
**Extrinsic state** — flyweight state that depends on context and therefore cannot be shared; passed in at call time. (Flyweight)
**Framework** — cooperating classes forming a reusable design for a class of software; provides architectural guidance and inverts control. Customized by subclassing and composing. (Ch 1, Ch 6, Template Method)
**Friend class** — C++ class granted the same access rights as the class itself. (Ch 1, Memento, Iterator)
**Hollywood Principle** — "Don't call us, we'll call you": the parent class calls subclass operations, not the reverse. (Template Method)
**Hook operation** — a Template Method step with a default (often empty) implementation that subclasses *may* override. Contrast **abstract operation**, which they *must*. (Template Method)
**Inheritance** — defines one entity in terms of another. *Class inheritance* combines *interface inheritance* (new interface from existing ones) and *implementation inheritance* (new implementation from existing ones). (Ch 1)
**Instance variable** — data defining part of an object's representation; C++ *data member*. (Ch 1)
**Interaction diagram** — shows the flow of requests between objects over time. (Ch 1, App B)
**Interface** — the set of all signatures defined by an object's operations; what requests it can respond to. (Ch 1)
**Intrinsic state** — flyweight state independent of context and therefore shareable. (Flyweight)
**Metaclass** — in Smalltalk, the class of a class object. (Ch 1, Abstract Factory, Prototype)
**Mixin class** — designed to be combined with others via (usually multiple) inheritance; normally abstract. (Ch 1)
**Narrow / wide interface** — a Memento offers a narrow interface to the Caretaker (opaque) and a wide one to the Originator (full state access). (Memento)
**Object** — a run-time entity packaging data and the procedures that operate on it. (Ch 1)
**Object composition** — assembling objects to get more complex behavior; the basis of black-box reuse. (Ch 1)
**Object diagram** — depicts a particular run-time object structure. (Ch 1, App B)
**Object reference** — a value identifying another object. (Ch 1)
**Operation** — the only way to manipulate an object's data; performed on receipt of a request. C++ *member function*, Smalltalk *method*. (Ch 1)
**Overriding** — redefining an inherited operation in a subclass. (Ch 1)
**Parameterized type** — leaves constituent types unspecified, supplied at point of use; C++ *templates*. A third way (besides inheritance and composition) to compose behavior. (Ch 1)
**Parent class** — the class another inherits from. Synonyms: *superclass* (Smalltalk), *base class* (C++), *ancestor class*. (Ch 1)
**Polymorphism** — substituting objects of matching interface for one another at run-time. (Ch 1)
**Private inheritance** — C++ inheritance solely for implementation, not interface. (Ch 1, Adapter)
**Protocol** — an interface extended to include the *allowable sequences* of requests. (Ch 1)
**Prototype (as mechanism)** — an existing instance cloned to produce new objects, replacing subclass-per-product. (Prototype, Abstract Factory)
**Receiver** — the target object of a request. (Ch 1, Command, Chain of Responsibility)
**Request** — what an object receives to trigger an operation; synonym *message*. (Ch 1)
**Robust iterator** — one that keeps traversal valid across insertions and removals in the aggregate. (Iterator)
**Signature** — an operation's name, parameters, and return value. (Ch 1)
**Subclass** — a class that inherits from another; C++ *derived class*. (Ch 1)
**Subsystem** — an independent group of classes collaborating to fulfill a set of responsibilities. (Ch 1, Facade)
**Subtype / Supertype** — a type is a subtype of another if its interface contains the other's; the parent type is the supertype. (Ch 1)
**Toolkit** — a collection of classes providing useful functionality without defining an application's design. Contrast **framework**, which does. (Ch 1)
**Type** — the name of a particular interface. Distinct from *class*, which also fixes implementation. (Ch 1)
**White-box reuse** — reuse via class inheritance; the subclass may see otherwise private aspects of the parent. (Ch 1, Ch 6)

## Classification vocabulary
**Purpose** — what the pattern does: **Creational**, **Structural**, or **Behavioral**. (Ch 1)
**Scope** — what the pattern applies to: **Class** (fixed at compile time via inheritance) or **Object** (changeable at run time via composition). (Ch 1)
