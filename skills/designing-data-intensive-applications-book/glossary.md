# Glossary — Designing Data-Intensive Applications (2nd ed.)

Definitions follow the authors' own usage. Chapter refs point to the fullest treatment.

**ACID** — Atomicity, Consistency, Isolation, Durability; coined 1983, vague enough that the marketing term "ACID compliant" says little (Ch 8)
**Anti-entropy** — background process copying missing data between replicas (Ch 6)
**Asynchronous** — not waiting for completion, making no assumption about how long it takes (Ch 6, 9)
**Atomic** — appears to take effect at a single point in time (concurrency); all-or-nothing group of writes (transactions) (Ch 8)
**Availability zone** — co-located resources likely to fail together; a region contains several (Ch 2, 6)
**Backpressure (flow control)** — forcing a sender to slow down when the recipient can't keep up (Ch 2, 12)
**Bloom filter** — probabilistic set-membership test; ~10 bits/key ≈ 1% false positives (Ch 4)
**Bounded / unbounded** — having / not having a known upper limit (delay, dataset size) (Ch 9, 12)
**Byzantine fault** — a node behaving arbitrarily or maliciously, sending contradictory messages (Ch 9)
**CAP theorem** — a widely misunderstood result that is not useful in practice; prefer PACELC (Ch 10)
**Causality / happens-before** — A happens before B if B knows about, builds on, or responds to A; neither ⇒ concurrent (Ch 6, 10)
**Change data capture (CDC)** — extracting all database writes as a stream so derived systems follow one leader (Ch 12)
**Clustered index** — the row stored inside the index; vs. **heap file** (unordered rows) and **covering index** (Ch 4)
**Compaction** — merging SSTables to discard overwritten values and tombstones; **size-tiered** vs. **leveled** (Ch 4)
**Consensus** — getting several nodes to agree on something; far harder than it looks (Ch 10)
**Consistent hashing** — hashing that keeps shards balanced while moving as few keys as possible on resize (Ch 7)
**Consistent prefix reads** — writes are observed in causal order (Ch 6)
**CRDT** — conflict-free replicated data type; automatic merge of concurrent writes (Ch 6)
**Crypto-shredding** — deleting immutable data by discarding its encryption key (Ch 3, 12)
**Data lake / lakehouse** — raw files with no imposed schema / query engine over object storage with table formats (Ch 1, 11)
**Data warehouse** — separate read-only copy of OLTP data prepared for analytics (Ch 1)
**Dead letter queue (DLQ)** — parking spot for poison messages; any entry is an error (Ch 12)
**Declarative** — specify the properties wanted, not the steps to get them (Ch 3)
**Denormalize** — introduce redundancy to speed reads; a precomputed query result (Ch 3)
**Derived data** — a dataset produced from other data by a repeatable process (index, cache, materialized view) (Ch 1, 13)
**Deterministic** — same input always yields same output; no clocks, randomness, or network (Ch 11)
**Dirty read / dirty write** — seeing / overwriting another transaction's uncommitted writes (Ch 8)
**Disaggregation** — separating storage and compute into independent services (Ch 1)
**Encoding (serialization)** — in-memory objects → self-contained byte sequence (Ch 5)
**Epoch number (ballot / view / term)** — monotonically increasing election number guaranteeing one leader per epoch (Ch 10)
**Eventual consistency** — replicas converge once writes stop; "eventually" is deliberately unbounded (Ch 6)
**Event time vs. processing time** — the timestamp in the event vs. the processor's clock; conflating them fabricates spikes (Ch 12)
**Exactly-once (effectively-once)** — the final effect equals a fault-free run, via idempotence plus fencing (Ch 5, 12, 13)
**Failover** — moving the leadership role from one node to another (Ch 6)
**Fault vs. failure** — a component deviating from spec vs. the system as a whole stopping service (Ch 2)
**Fencing token** — monotonic token proving lock currency; the defense against zombie leaseholders (Ch 9)
**FLP result** — no deterministic asynchronous algorithm always terminates if a node may crash (Ch 10)
**Follower** — replica that only applies changes received from a leader (Ch 6)
**Head-of-line blocking** — a few slow requests holding up fast ones queued behind them (Ch 2)
**HTAP** — one system serving both low-latency record access and large scans (Ch 1)
**Idempotent** — safe to retry; executing more than once has the same effect as once (Ch 5, 12)
**Isolation** — degree to which concurrent transactions interfere; **serializable** is the strongest (Ch 8)
**Kappa architecture** — historical reprocessing and live streaming in one system (Ch 13)
**Lamport clock** — logical timestamp `(counter, node ID)` giving a total order consistent with causality (Ch 10)
**Lease** — a lock with a timeout that must be periodically renewed (Ch 9)
**Linearizable** — behaves as if a single copy of the data exists, updated by atomic operations (Ch 10)
**Log** — append-only file; write-ahead log, log-structured storage, replication log, or event log (Ch 4, 6, 12)
**Log compaction** — retaining only the latest value per key, so disk use tracks current contents not write history (Ch 12)
**LSM-tree / SSTable / memtable** — log-structured storage engine / immutable key-sorted file / in-memory write buffer (Ch 4)
**Lost update** — concurrent read-modify-write cycles where the later write clobbers the earlier (Ch 8)
**Materialize** — compute eagerly and write out the result rather than on demand (Ch 2, 3)
**Metastable failure** — a retry storm keeping a system broken even after load drops (Ch 2)
**Monotonic reads** — never seeing older data after having seen newer (Ch 6)
**MVCC** — several committed row versions side by side so transactions see different points in time (Ch 8)
**N+1 query problem** — one query per row instead of a single join (Ch 3)
**OLTP / OLAP** — fast indexed access to few records / aggregation over many records (Ch 1)
**Outbox pattern** — CDC table written in the same transaction as domain data (Ch 12)
**Percentile** — value below which a given share of measurements fall; p95, p99 describe tails (Ch 2)
**Quorum (n, w, r)** — replicas, write acks, read queries; overlap requires `w + r > n`; **sloppy quorum** accepts any reachable replica (Ch 6)
**Read-after-write (read-your-writes)** — a user always sees their own submitted updates (Ch 6)
**Read repair / hinted handoff** — leaderless catch-up on read / after a node returns (Ch 6)
**Rebalance** — move data or services between nodes to spread load fairly (Ch 7)
**Response time vs. service time vs. queueing delay** — what the client sees / active processing / waiting (Ch 2)
**Rolling upgrade** — patching one node at a time with no downtime; the reason compatibility matters (Ch 2, 5)
**Schema evolution** — changing a schema while old and new code keep interoperating; **backward** = new code reads old data, **forward** = old code reads new data (Ch 5)
**Schema registry** — service storing schema versions and checking compatibility before deploy (Ch 5)
**Schema-on-read vs. schema-on-write** — structure interpreted at read time vs. enforced at write time (Ch 3)
**Secondary index** — extra structure for finding records matching a condition; **local/document-partitioned** vs. **global/term-partitioned** (Ch 4, 7)
**Serializable snapshot isolation (SSI)** — optimistic full serializability at small cost over snapshot isolation (Ch 8)
**Shard (partition)** — subset of data; every record belongs to exactly one (Ch 7)
**Shared-nothing** — independent nodes with own CPU, memory, disk, connected by a network (Ch 1, 2)
**Shuffle** — distributed sort where mappers write one hash-partitioned sorted file per reducer (Ch 11)
**Skew** — imbalanced load across shards (hot spots); also a timing anomaly (read/write/clock skew) (Ch 7, 8, 9)
**Snapshot isolation** — each transaction reads from a consistent snapshot as of its start (Ch 8)
**Split brain** — two nodes simultaneously believing they are leader (Ch 6, 9)
**State machine replication** — apply the same deterministic operation log in the same order on every replica (Ch 10)
**Stored procedure** — whole transaction submitted ahead of time, avoiding per-statement round trips (Ch 8)
**System of record** — holds the primary authoritative data; the source of truth (Ch 1, 13)
**Tail latency amplification** — many backend calls per user request ⇒ more slow user requests (Ch 2)
**Timeliness vs. integrity** — users see up-to-date state (violations temporary) vs. data is uncorrupted and correctly derived (violations perpetual) (Ch 13)
**Timeout** — the only general way to detect a fault; can never distinguish node from network failure (Ch 9)
**Tombstone** — deletion marker appended so compaction can discard older values (Ch 4)
**Total order broadcast** — same messages delivered to all nodes in the same order; equivalent to consensus (Ch 10)
**Two-phase commit (2PC)** — coordinator-driven atomic commit across nodes; participants can be left **in doubt** (Ch 8)
**Two-phase locking (2PL)** — serializability by holding read/write locks until transaction end (Ch 8)
**Version vector** — per-replica version numbers distinguishing overwrites from concurrent writes (Ch 6)
**Vector embedding** — float array locating a document in semantic space; indexed by HNSW or IVF (Ch 4)
**Windowing** — **tumbling** (fixed, non-overlapping), **hopping** (fixed, overlapping), **sliding** (interval-relative), **session** (inactivity-delimited) (Ch 12)
**Write amplification** — bytes actually written to disk ÷ bytes a plain append-only log would write (Ch 4)
**Write skew / phantom** — transactions read the same objects then update different ones, breaking an invariant / a write changes another transaction's search result (Ch 8)
**Zombie** — a former leaseholder still acting as holder after losing the lease (Ch 9)
