# Chapter 22: The Clean Architecture

## Core Idea
Integrate Hexagonal Architecture, DCI, and BCE into one actionable idea: concentric circles of software — **Entities, Use Cases, Interface Adapters, Frameworks and Drivers** — governed by a single overriding rule, **the Dependency Rule**: *Source code dependencies must point only inward, toward higher-level policies.*

## Frameworks Introduced
- **The Dependency Rule**: *Source code dependencies must point only inward, toward higher-level policies.*
  - When to use: at every source-code dependency in the system, without exception; it applies no matter how many circles you end up with.
  - How:
    1. Arrange software into concentric circles. Outer circles are **mechanisms**; inner circles are **policies**. The further in you go, the higher level the software becomes.
    2. Nothing in an inner circle may know anything at all about something in an outer circle. In particular, **the name of something declared in an outer circle must not be mentioned by the code in an inner circle** — functions, classes, variables, or any other named software entity.
    3. The same prohibition covers **data formats** declared in an outer circle, especially formats generated by a framework. Nothing in an outer circle may impact the inner circles.
    4. Where the flow of control must run outward, invert the dependency (see below) rather than weakening the rule.
  - Why it works: it makes the system intrinsically testable, and lets you replace obsolete external parts — the database, the web framework — with a minimum of fuss, because no inner code names them.
  - Failure mode: importing a framework's DTO/row type "just for convenience" into the use case layer. It compiles, and it silently binds your highest-level policy to the outermost detail.

- **The four concentric circles**: Entities → Use Cases → Interface Adapters → Frameworks and Drivers (innermost to outermost).
  - When to use: as a schematic. There is no rule that says you must always have just these four; you may need more. The Dependency Rule always applies regardless of count.
  - How: place a module by asking how general its policy is. Innermost is the most general and highest level; outermost is low-level concrete detail.

- **Crossing boundaries via the Dependency Inversion Principle**: use dynamic polymorphism to create source code dependencies that *oppose* the flow of control.
  - When to use: whenever flow of control must move outward — a use case that needs to call a presenter, a gateway, or an external service.
  - How:
    1. Declare an interface in the inner circle — for output, the **use case output port**.
    2. Have the outer-circle class (the Presenter) implement it.
    3. The use case calls the interface, never the presenter by name. Flow of control goes outward; the source code dependency points inward.
    4. Apply the identical technique at every boundary, in both directions.

- **Which data crosses the boundaries**: only isolated, simple data structures.
  - How: use basic structs, simple data transfer objects, plain arguments in function calls, a hashmap, or a constructed object. Never pass Entity objects or database rows. When data crosses a boundary, it is always in the form **most convenient for the inner circle**.

## Key Concepts
- **Policy vs. mechanism**: inner circles are policies (high-level rules); outer circles are mechanisms (concrete details).
- **Entities**: encapsulate **enterprise-wide Critical Business Rules**; an object with methods or a set of data structures and functions, usable by many different applications in the enterprise.
- **Use Cases**: **application-specific** business rules that orchestrate the flow of data to and from the entities and direct them to use their Critical Business Rules.
- **Interface Adapters**: adapters converting data between the form most convenient for use cases/entities and the form most convenient for an external agency (database, web, external service). MVC lives wholly here.
- **Frameworks and Drivers**: the outermost layer — database, web framework, tools. Mostly glue code. "This is where all the details go."
- **Use case output port / input boundary**: inner-circle interfaces that let control flow outward without an outward source dependency.
- **Row structure**: the convenient data format a database framework returns from a query — must never be passed inward across a boundary.
- **ViewModel**: a plain old Java object of mostly Strings and flags, produced by the Presenter, that the View merely moves onto the screen.

## Mental Models
- Think of the circles as an **altitude map**: as you move inward, abstraction and policy increase; the innermost circle is the most general and highest level.
- Use the **name test** as the mechanical check for the Dependency Rule: grep an inner module for any identifier declared outward. One hit is a violation.
- Think of a boundary crossing as **flow of control and dependency pointing opposite ways** — polymorphism is the pivot that lets them disagree.
- Treat "the web is a detail, the database is a detail" as a **placement rule**, not a slogan: details go outside where they can do little harm.

## Reference Tables

**The four circles**

| Circle (in → out) | Contains | Changes when | May name | Must never name |
|---|---|---|---|---|
| **Entities** | Enterprise-wide Critical Business Rules; objects with methods or data structures + functions | Almost never; not by page navigation or security changes; no operational change to any particular application should affect it | Nothing outside itself | Use cases, adapters, frameworks, DB |
| **Use Cases** | Application-specific business rules; the system's use cases; orchestration of Entities | The operation/details of a use case change | Entities | UI, database, frameworks, presenters |
| **Interface Adapters** | Controllers, Presenters, Views (the whole MVC); gateways' implementations; SQL; format conversion | The external agency's format changes | Use Cases, Entities | Framework internals it doesn't glue to |
| **Frameworks and Drivers** | Database, web framework, tools; glue code only | The tool or vendor changes | Everything inward | — |

**Boundary crossing anatomy**

| Element | Where it lives | Direction of control | Direction of source dependency |
|---|---|---|---|
| Controller | Interface Adapters | inward (calls) | inward |
| InputBoundary (interface) | Use Cases | — | — |
| UseCaseInteractor | Use Cases | outward to Presenter | — |
| OutputBoundary / output port (interface) | Use Cases | — | Presenter depends inward on it |
| Presenter | Interface Adapters | receives control | inward |
| DataAccessInterface | Use Cases | — | Database implementation depends inward on it |

## Worked Example
**One request through a web-based Java system using a database (the typical scenario).**

1. The **web server** gathers input data from the user and hands it to the **Controller** (upper left, Interface Adapters).
2. The **Controller** packages that data into a plain old Java object and passes it through the **InputBoundary** interface to the **UseCaseInteractor**. The InputBoundary is declared in the use case layer, so the Controller's source dependency points inward.
3. The **UseCaseInteractor** interprets the data and uses it to control the dance of the **Entities**. To bring the data those Entities need into memory, it calls the **DataAccessInterface** — again declared inward, implemented by a class in the **Database** layer. No SQL appears inside the interactor; all SQL is restricted to the database part of the Interface Adapters layer.
4. On completion, the UseCaseInteractor gathers data from the Entities and constructs **OutputData**, another plain old Java object — not an Entity, not a row structure.
5. OutputData is passed through the **OutputBoundary** interface to the **Presenter**. Control flows outward; the dependency points inward, because the Presenter implements an interface owned by the use case layer.
6. The **Presenter** repackages OutputData into viewable form as the **ViewModel**, yet another plain old Java object of mostly Strings and flags. Where OutputData held a `Date`, the ViewModel holds a String already formatted for the user; the same for `Currency` and any other business data. Button and MenuItem names go into the ViewModel, along with flags telling the View whether they should be gray.
7. The **View** has almost nothing left to do but move data from the ViewModel into the HTML page.

Every arrow crossing a boundary line points inward. That is the whole trick.

## Key Takeaways
1. Memorize the rule verbatim: *Source code dependencies must point only inward, toward higher-level policies.*
2. An inner circle may not mention the *name* of anything in an outer circle — nor use data formats declared there, especially framework-generated ones.
3. Four circles are schematic, not mandatory; add circles freely, but never relax the Dependency Rule.
4. When control must flow outward, invert the dependency with a polymorphic port declared in the inner circle.
5. Pass only isolated, simple data structures across boundaries — never Entity objects, never database rows — always in the form most convenient for the inner circle.
6. Restrict all SQL to the database portion of the Interface Adapters layer; no code inward of that circle knows the database exists.
7. The payoff is a system independent of frameworks, testable without UI/DB/web server, independent of the UI, independent of the database, and independent of any external agency.
