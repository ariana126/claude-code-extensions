# Glossary — Fundamentals of Software Architecture (2nd Ed.)

**Abstractness** — ratio of abstract artifacts to all artifacts: `A = Σmₐ / (Σmₐ + Σm_c)` (Ch 3)
**Accidental complexity** — the code/design is more complex than it needs to be; "we made a problem hard" (Ch 3, Ch 6, Ch 25)
**Accidental SOA** — antipattern where an ESB gradually encapsulates the entire architecture (Ch 17)
**ADR (Architectural Decision Record)** — 1–2 page record: Title, Status, Context, Decision, Consequences, + Compliance and Notes (Ch 21)
**Afferent coupling (CA)** — incoming/fan-in: how many components depend on this one (Ch 3, Ch 8)
**Ambulance pattern** — separate message channels to prioritize urgent requests over ordinary ones (Ch 22)
**AMQP** — Advanced Message Queuing Protocol; events publish to an exchange that forwards to per-subscriber queues (Ch 15)
**Anemic event** — derived event whose payload lacks the context needed for downstream processing (Ch 15)
**Antipattern** — something that seems like a good idea when you begin but leads you into trouble (Andrew Koenig) (Ch 2, Ch 21)
**API layer / API Gateway** — routing plus cross-cutting concerns between consumers and services; **never** a mediator or orchestrator in microservices (Ch 14, Ch 18)
**ArchiMate** — The Open Group's lightweight open source enterprise-architecture modeling language (Ch 23)
**Architecture by Implication** — antipattern of coding without knowing which style you're using (Ch 10)
**Architecture characteristics** — the "-ilities"; a system's *capabilities*, as opposed to the domain's *behavior* (Ch 1, Ch 4)
**Architecture decision** — a rule constraining how a system may be constructed (Ch 1, Ch 21)
**Architecture isomorphism** — the generic "shape" of an architecture: how components depend on each other in the topology (Ch 19)
**Architecture quantum** — the smallest part of a system that runs independently: independently deployable, high functional cohesion, low external implementation static coupling, synchronous communication with other quanta (Ch 7)
**Architecture Sinkhole** — antipattern where requests pass through layers with no business logic performed; concerning above ~20% of requests (Ch 10)
**Architecture style** — a named topology defined by component topology, physical architecture, deployment, communication style, and data topology (Ch 9)
**Architecture vitality** — how viable an architecture defined 3+ years ago still is today (Ch 1)
**Architecturally significant** — Nygard's term: decisions affecting structure, non-functional characteristics, dependencies, interfaces, or construction techniques (Ch 21)
**Armchair architect** — hasn't coded in a long time, disconnected from the team, creates boundaries that are too loose (Ch 24)
**Backends for Frontends (BFF)** — thin adapters translating generic backend output into device-specific format (Ch 19)
**BASE transactions** — basic availability, soft state, eventual consistency; the distributed alternative to ACID (Ch 14)
**Big Ball of Distributed Mud** — the distributed form of Big Ball of Mud, caused by granularity failures (Ch 3, Ch 18)
**Big Ball of Mud** — haphazardly structured, sprawling, sloppy, duct-tape-and-baling-wire spaghetti-code jungle (Foote & Yoder, 1997) (Ch 9)
**Bottleneck Trap** — antipattern where the architect owns critical-path code and blocks the team (Ch 2)
**Bounded context** — DDD region where everything about a portion of the domain is visible internally but opaque outside (Ch 7, Ch 18)
**Broker** — the software product containing queues and topics; implements event handlers (Ch 15, Ch 20)
**Brooks's Law** — adding people to a late project makes it later; actual productivity always trails group potential (Ch 24)
**Business Delegate** — pattern reducing coupling between business services and the UI (Ch 10)
**Business services** — SOA's coarse-grained top layer: no code, just input/output/schema, defined by business analysts (Ch 17)
**C4** — Simon Brown's diagramming technique: Context, Container, Component, Class (Ch 23)
**Choreography** — no central coordinator; services call each other as needed (Ch 15, Ch 18, Ch 20)
**Client acknowledge mode** — messaging mode keeping an event in the queue until the consumer confirms processing (Ch 15)
**Cohesion** — the extent to which a module's parts belong together. Scale (best→worst): functional, sequential, communicational, procedural, temporal, logical, coincidental (Ch 3)
**Compensating transaction / update** — orchestrator-issued reversal restoring prior state when part of a distributed transaction fails (Ch 9, Ch 18)
**Composite architectural characteristic** — one with no single objective definition, composed of measurable parts; e.g. agility = deployability + modularity + testability (Ch 5)
**Connascence** — Page-Jones's precise language for coupling: two components are connascent if changing one requires changing the other (Ch 3)
**Connascence of Name / Type / Meaning / Position / Algorithm** — static connascence, weakest to strongest (Ch 3)
**Connascence of Execution / Timing / Values / Identity** — dynamic connascence, weakest to strongest (Ch 3)
**Control-freak architect** — controls every detail, creates boundaries that are too tight (Ch 24)
**Conway's Law** — organizations produce designs copying their own communication structures (Ch 9)
**Correlation ID (CID)** — reply-message header field set to the original request's message ID; the recommended request-reply technique (Ch 15)
**Covering Your Assets** — antipattern of avoiding or deferring a decision out of fear (Ch 21)
**CQRS** — Command-Query-Responsibility-Segregation: isolate writes into one datastore that synchronizes to a separate read datastore (Ch 20)
**Cyclic dependencies** — components referencing each other in a network; you can't reuse one without the others (Ch 6)
**Cyclomatic Complexity (CC)** — McCabe, 1976: `CC = E − N + 2P`. Industry threshold <10; the authors prefer <5 (Ch 6)
**Data collision** — replicated-cache inconsistency when the update rate exceeds replication latency. `Collision Rate = (N × UR²) / S × RL` (Ch 16)
**Data grid** — the component managing synchronization and replication between processing units (Ch 16)
**Data pump** — always-asynchronous mechanism sending cache updates to the database (Ch 16)
**Data-based event payload** — event carrying all information needed for processing (Ch 15)
**Database-per-Service** — the standard microservices data topology (Ch 18)
**Dedicated database topology** — one database per event processor; highest fault tolerance, scalability, change control (Ch 15)
**Deployment manager** — starts and tears down processing-unit instances as load changes; critical to elasticity (Ch 16)
**Derived event** — event triggered by a processor after handling an initiating event (Ch 15)
**Diffusion of responsibility** — as teams grow, ownership becomes unclear and work gets dropped (Ch 24)
**Distance from the Main Sequence** — `D = |A + I − 1|`; the one holistic structural metric (Ch 3)
**Distributed architecture** — multiple deployment units connected by remote access protocols (Ch 9)
**Distributed caching** — external centralized cache; optimizes consistency over performance and fault tolerance (Ch 16)
**DLL Hell / JAR Hell** — antipattern from excessive compile-time dependencies between modules (Ch 11)
**Domain-Broker pattern** — one broker per domain group; better isolation and scalability, harder discovery (Ch 20)
**Domain database topology** — one database per domain group of event processors (Ch 15)
**Domain partitioning** — organizing top-level components by domain/workflow rather than technical capability (Ch 9)
**Domain service** — a coarse-grained, separately deployed service representing a domain or subdomain (Ch 14)
**Durable subscriber** — a subscriber guaranteed to receive an event; the topic stores it while the subscriber is down (Ch 15)
**Dynamic coupling** — the forces involved when quanta communicate at runtime (Ch 7)
**Dynamic Quantum Entanglement** — antipattern where two quanta communicating synchronously collapse into one (Ch 15)
**Efferent coupling (CE)** — outgoing/fan-out: how many components this one depends on (Ch 3, Ch 8)
**Elastic Leadership** — calibrating architect involvement to team familiarity, size, experience, project complexity, and duration (Ch 24)
**Elasticity** — withstanding *bursts* of traffic (distinct from scalability) (Ch 5)
**Email-Driven Architecture** — antipattern where decisions are lost, forgotten, or never received (Ch 21)
**Enterprise services** — SOA's fine-grained shared implementations layer (Ch 17)
**Entity Trap** — antipattern of deriving components from entities; tells: *Manager*, *Supervisor*, *Controller*, *Handler*, *Engine*, *Processor* suffixes (Ch 8)
**Essential complexity** — the code is complex because the problem is complex (Ch 3, Ch 6, Ch 25)
**Event** — a broadcast that something has already happened; typically needs no response; uses topics/streams (Ch 15)
**Event broker** — usually federated; contains the event channels for a domain's event flow (Ch 15)
**Event Forwarding pattern** — technique for preventing data loss in AMQP/JMS event channels (Ch 15)
**Event mediator** — orchestrator that knows the workflow steps and issues derived messages point-to-point (Ch 15)
**Extensible derived event** — an event nobody currently consumes, providing a hook for future functionality (Ch 15)
**Fallacies of distributed computing** — Deutsch et al., 1994, plus three from the authors (Ch 9)
**Fast-Lane Reader** — early-2000s pattern letting Presentation hit the database directly; destroys layers of isolation (Ch 10)
**Filter** — pipeline component performing one task; types: producer, transformer, tester, consumer (Ch 12)
**Fitness function** — any mechanism providing an objective integrity assessment of an architecture characteristic (Ch 6)
**Flow** — state where the brain is 100% engaged and hours feel like minutes; don't disrupt it (Ch 25)
**Front Controller** — antipattern where a nominally choreographed service becomes a complex mediator (Ch 18)
**Frozen Caveman** — antipattern where an architect reverts to a pet irrational concern on every architecture (Ch 2)
**Functional cohesion** — the semantic binding created by business concepts (Ch 7)
**Generic architecture** — antipattern of designing to support *all* characteristics (Ch 5)
**Governance** — from Greek *kubernan*, "to steer"; any aspect of development the architect must influence (Ch 6)
**Grains of Sand** — antipattern of making microservices too fine-grained (Mark Richards, 2016) (Ch 18)
**Granularity** — the *size* of a decomposed piece, as opposed to modularity (Ch 3)
**Groundhog Day** — antipattern where a decision is re-litigated forever because its *why* was never given (Ch 21)
**Hawthorne effect** — people change behavior when they believe they're being observed (Ch 24)
**Hexagonal architecture (Ports and Adapters)** — Cockburn's pattern; treats the database as a pluggable adapter, which conflicts with microservices bounded contexts (Ch 20)
**Implementation coupling** — how the team chooses to implement dependencies; fully under the architect's control (Ch 7)
**Implicit characteristic** — necessary but never stated in requirements; derived from domain knowledge (Ch 4)
**Initiating event** — the event that starts an entire event flow (Ch 15)
**Instability** — `I = Ce / (Ce + Ca)`; measures volatility (Ch 3)
**Inverse Conway Maneuver** — evolving team structures to promote the desired architecture (Ch 9)
**Irrational Artifact Attachment** — attachment to an artifact grows with the time spent producing it (Ch 23)
**Ivory Tower architecture** — antipattern of deciding in isolation from the implementation team (Ch 5, Ch 25)
**Kata (architecture kata)** — timeboxed practice exercise deriving characteristics from a domain description (Ted Neward) (Ch 5)
**Key-based event payload** — event carrying only a context key; responders query the database (Ch 15)
**K-weight budget** — maximum bytes of libraries and frameworks allowed on a page (Ch 6)
**Last Participant Support (LPS)** — removes an event from the persisted queue only after confirming full processing and persistence (Ch 15)
**Last responsible moment** — decide when the cost of deferring exceeds the risk of deciding (Ch 21)
**Layers of isolation** — changes in one layer don't affect others, provided contracts hold and intervening layers are closed (Ch 10)
**LCOM (Lack of Cohesion in Methods)** — Chidamber & Kemerer; "the sum of sets of methods not shared via sharing fields" (Ch 3)
**Least worst architecture** — the goal; never strive for the *best* (Ch 4)
**Logical architecture** — the system's logical components and their interactions, independent of physical deployment (Ch 8)
**Logical component** — a building block implementing a business function; a namespace or directory leaf node (Ch 8)
**Mediator (modular monolith)** — component decoupling modules by acting as an orchestrator (Ch 11)
**Messaging grid** — routes input requests to available processing units and manages session state (Ch 16)
**Message** — a command or query; usually needs a response; directed to one consumer; uses queues (Ch 15)
**Micro-frontends** — UI-level components matching backend service granularity (Ch 18)
**Modularity** — a logical grouping of related code; an implicit architecture characteristic (Ch 3)
**Module** — in modular monolith, a domain or subdomain composed of one to many components (Ch 11)
**Monolithic architecture** — a single deployment unit for all code (Ch 9)
**Near-cache** — hybrid front cache + full backing cache; **not recommended** in space-based architecture (Ch 16)
**Non-functional requirements** — the rejected legacy term for architecture characteristics; self-denigrating (Ch 4)
**Orchestration** — a central coordinator manages the workflow (Ch 15, Ch 18, Ch 20)
**Orthogonal coupling** — two parts with distinct purposes that must still intersect (Ch 20)
**Out of Context antipattern** — understanding the trade-offs but failing to *weight* them for the current context (Ch 27)
**Persisted queue** — queue storing messages on disk as well as in memory; enables guaranteed delivery (Ch 15, Ch 16)
**Physical architecture** — services, UIs, databases, and other deployment artifacts (Ch 8)
**Pluralistic ignorance** — everyone privately rejects a norm but publicly agrees, fearing they've missed something obvious (Ch 24)
**Poison event** — a derived event that keeps looping forever between services (Ch 15)
**Processing unit** — space-based component containing application logic plus the in-memory data grid (Ch 16)
**Process loss** — the gap between a team's potential and actual productivity (Ch 24)
**Protocol-aware heterogeneous interoperability** — microservices communication: services know the protocol, stacks may differ, services call each other (Ch 18)
**Replicated caching** — each unit holds a synchronized in-memory grid; optimizes performance and fault tolerance (Ch 16)
**Replication latency (RL)** — time to synchronize a same-named cache; use 100 ms as a planning default (Ch 16)
**Representational consistency** — always showing relationships between parts before changing views (Ch 23)
**Request-reply (pseudosynchronous)** — two-queue pattern for when a response is needed immediately (Ch 15)
**Residuality theory** — Barry O'Reilly: treat business change as *stressors* and architectural change as *residues* (Ch 26)
**Risk storming** — collaborative three-phase exercise (identification, consensus, mitigation) for architecture risk (Ch 22)
**Sacrificial architecture** — deliberately simple architecture expected to be replaced (Ch 5)
**Saga** — distributed transaction pattern using a mediator plus compensating transactions (Ch 18)
**Scalability** — handling growing concurrent users over time (distinct from elasticity) (Ch 5)
**Semantic coupling** — the natural coupling of the problem domain; the architect has almost no leverage over it (Ch 7)
**Semantic decoupling** — one event processor having no knowledge of or dependency on another's actions (Ch 15)
**Service discovery** — automatically detecting and locating services; the elasticity mechanism in microservices (Ch 18)
**Service locator pattern** — embedded in the UI or gateway so it can reach services directly (Ch 14)
**Service mesh** — sidecars connected by a service plane (e.g. Istio) giving unified operational control (Ch 18, Ch 20)
**Sidecar** — component holding cross-cutting operational concerns inside each service (Ch 18, Ch 20)
**SLA / SLO** — service-level agreement (usually legally binding) / objective (usually not) (Ch 22)
**Space-based architecture** — style replacing the central database with replicated in-memory data grids (Ch 16)
**Stale expertise** — the mistaken sensation that outdated information is still cutting edge (Ch 2)
**Stamp coupling** — passing far more data than the consumer needs (Ch 9, Ch 15)
**Static coupling** — the "wiring" of an architecture; how services depend on one another (Ch 7)
**Strangler Pattern** — incremental migration technique supported well by Agile processes (Ch 26)
**Structural decay** — degradation from changes that erode required architecture characteristics (Ch 1)
**Superseded (ADR status)** — a decision replaced by another; always assumes the prior status was Accepted (Ch 21)
**Swarm of Gnats** — antipattern of triggering too many fine-grained derived events (Ch 15)
**Team topologies** — Skelton & Pais: stream-aligned, enabling, complicated-subsystem, platform teams (Ch 9)
**Technical breadth** — knowing a little about a lot; the architect's primary knowledge asset (Ch 2)
**Technical depth** — deep knowledge of one thing; requires maintenance or it decays (Ch 2)
**Technical partitioning** — organizing top-level components by technical capability (Ch 9)
**Temporal coupling** — nonstatic dependencies based on timing or transactions; hard to detect with tooling (Ch 8)
**Top-level partitioning** — the choice between technical and domain organization; one of the first architect decisions (Ch 9)
**Ubiquitous language** — DDD practice of agreeing on objective shared definitions (Ch 3, Ch 4, Ch 6)
**Unknown unknowns** — the things nobody knew would arise; why Big Design Up Front fails (Ch 26)
**Unstructured monolith** — a monolith with code so interdependent it cannot be unraveled (Ch 11)
**Virtualized middleware** — space-based infrastructure managing and coordinating processing units (Ch 16)
**Workflow Event pattern** — reactive error handling via delegation, containment, and repair (Ch 15)
**Zone of Pain / Zone of Uselessness** — too much implementation (brittle) / too much abstraction (unusable) (Ch 3)
**20-minute rule** — spend at least 20 minutes daily learning, *before checking email* (Ch 2)
